Scene 1 · Authentication

Secure Login with Role-Based Redirect

🔐
Any Staff User
Admin · Custodian · Auditor
auth
Route: /login
The login page is the only public route. All other routes are protected by middleware — unauthenticated requests are silently redirected to /login.
Actions
1
Open /login — enter username admin and password password123
2
System validates credentials via bcrypt (12-round hash) — no plain-text passwords stored
3
On success: JWT session issued → redirect to /dashboard
4
Try wrong password — see the error message; page does not expose which field is wrong
5
Open an incognito tab and visit /dashboard directly — middleware redirects to /login
Security implementation
Passwords are hashed with bcryptjs (12 rounds). Sessions use JWT strategy via Auth.js v5. Middleware reads the session token from the cookie on every request — no server-side state required. All role data is embedded in the JWT token and verified server-side.

Scene 6 · Transactions

Custodian Records IN — Stock Received (Borrowable)

C
Supplies Custodian
Custodian · General Supplies Office
custodian
Module: Transactions · Type: IN
The first action on a newly added item is recording incoming stock. An IN transaction increases the available quantity and is the foundation of the ledger.
Transaction Type: IN
No requester required.
Stock formula: SUM(IN + RETURN) − SUM(OUT)
Actions
1
Navigate to Transactions → New Transaction
2
Select type: IN — requester field hidden (not required for IN)
3
Choose item: LCD Projector — Room A, quantity: 5, notes: "Received from procurement"
4
Submit → transaction is recorded; navigate back to Inventory
5
The item now shows Stock: 5 — computed live from the transaction log
Ledger-based stock
The stock figure is never a field you edit directly. It is always derived from the full transaction history — making it impossible to silently adjust stock without leaving a record. Every change is attributable to a specific user and timestamp.

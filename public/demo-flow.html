Scene 1 · Authentication

Secure Login with Role-Based Redirect

🔐
Any Staff User
Admin · Custodian · Auditor
auth
Route: /login
The login page is the only public route. All other routes are protected by middleware — unauthenticated requests are silently redirected to /login.
Actions
1
Open /login — enter username admin and password password123
2
System validates credentials via bcrypt (12-round hash) — no plain-text passwords stored
3
On success: JWT session issued → redirect to /dashboard
4
Try wrong password — see the error message; page does not expose which field is wrong
5
Open an incognito tab and visit /dashboard directly — middleware redirects to /login
Security implementation
Passwords are hashed with bcryptjs (12 rounds). Sessions use JWT strategy via Auth.js v5. Middleware reads the session token from the cookie on every request — no server-side state required. All role data is embedded in the JWT token and verified server-side.

Scene 3 · Inventory

Custodian Adds an Item — QR Code Auto-Generated

C
Equipment Custodian
Custodian · College of Criminology
custodian
Module: Inventory
Every item added to the system automatically gets its own QR code — no external tool required. The QR value is a unique string tied to the item's database ID.
QR value format
INV-<item-id>
e.g. INV-cm5ab3k2f0000xyz12345
Actions
1
Log in as custodian → navigate to Inventory
2
Click Add Item — fill in: name (e.g. Baton — Drill Training), description, category (Training Equipment), reorder level (10)
3
Click Save — item appears in the table; a QR code is generated in the background automatically
4
Click the item row to open the Item Detail page — the QR code image is displayed on the right
5
Click Download QR — saves a PNG file ready for printing and labeling the physical item
6
Note the current stock: 0 — stock is computed live from all IN/OUT/RETURN transactions; no manual stock field
Design highlight
Stock is never stored directly — it is computed from the transaction ledger every time. IN adds stock; OUT deducts it; RETURN adds it back. This makes the audit trail the single source of truth.

Scene 5 · Transactions

Custodian Records an IN Transaction (Stock Received)

C
Equipment Custodian
Custodian · College of Criminology
custodian
Module: Transactions · Type: IN
The first action on a newly added item is recording incoming stock. An IN transaction increases the available quantity and is the foundation of the ledger.
Transaction Type: IN
No borrower required.
Stock formula: SUM(IN + RETURN) − SUM(OUT)
Actions
1
Navigate to Transactions → New Transaction
2
Select type: IN — borrower field disappears (not required for IN)
3
Choose item: Baton — Drill Training, quantity: 30, notes: "Received from procurement — batch 1"
4
Submit → transaction is recorded; navigate back to Inventory
5
The item now shows Stock: 30 — computed live from the transaction log
Ledger-based stock
The stock figure is never a field you edit directly. It is always derived from the full transaction history — making it impossible to silently adjust stock without leaving a record. Every change is attributable to a specific user and timestamp.

Accountability on every OUT
An OUT transaction without a borrower is rejected by the system — the form will not submit. Every issuance is linked to the specific student who received the equipment, creating a borrower-accountable chain of custody.

Actions

2
Set date range to cover today — click Apply Filter

RBAC in action
Despite having read access to Reports and Audit Logs, the Auditor cannot create or modify any record. There is no "Add" button, no edit option, no delete — enforced both in the UI and at the API level via requireRole() server-side checks.
